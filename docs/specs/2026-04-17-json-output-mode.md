# Agent-friendly JSON output: `./hydra status --json` and `./hydra ps --json`

## Problem

Today `./hydra status` emits prose formatted for a human reading the terminal (ANSI colors, header/footer rules, "X enabled / Y total" strings). An agent driving Hydra — OpenClaw, the forthcoming MCP server binary from #72, or any external client — has to regex-parse that prose to answer basic questions like "how many workers are running?" or "is autopickup on?". The prose format is also not a stable contract: a minor UI tweak breaks every downstream parser silently.

The operator's directional ask: **make it agent-friendly to use and for development.** The rest of the `./hydra` CLI (`add-repo`, `pause`, `issue`) is already safe for scripting because each helper is `set -euo pipefail` with documented exit codes — only `status` emits free-form text. Closing that gap is the smallest change that unlocks programmatic orchestration, and it lays the foundation that `mcp/hydra-tools.json:hydra.get_status` already declares in its `output_schema` block.

There is no `./hydra ps` today — per-worker detail lives inline in `status`. Agents that want just "list my workers" still have to pay for the full snapshot and string-match rows out of it.

## Goals / non-goals

**Goals:**

- `./hydra status --json` prints a structured JSON object covering the same fields the prose view surfaces: active workers, tickets-today counters, autopickup block, pause state, quota health.
- `./hydra ps --json` prints a per-worker detail array (`{id, ticket, repo, tier, started_at, worktree, branch, status, minutes_running}`) — useful on its own, and a strict subset of `status --json:active_workers` semantics.
- One stable, additive-only JSON Schema at `state/schemas/hydra-status-output.schema.json`. Includes a top-level `version` field so downstream agents can version-gate their parsers.
- A self-test (`json-output-schema-valid`, kind `script`) that pipes `./hydra status --json` in a fixture through `jq empty` and (if available) `ajv validate`, asserting schema match on every CI run.
- A short README section "Driving Hydra from an agent" with a three-line recipe using `./hydra status --json | jq '.active_workers | length'` or equivalent.
- A cross-reference in `mcp/hydra-tools.json` (inside the existing `_description` or a new `_see_also` field) pointing at the new schema as the canonical shape. No tool-contract change — the MCP output schema already matches.

**Non-goals:**

- JSON mode on every subcommand. Only `status` and `ps` for v1. `pause`, `resume`, `add-repo`, etc. are side-effecting and already script-safe via exit codes.
- Streaming / watch mode (SSE, WebSocket, `--follow`). Snapshot-only — the agent polls on whatever cadence it wants.
- Breaking the existing prose output. `--json` is strictly opt-in. `./hydra status` with no flag renders the exact same ANSI text today's operators see.
- Re-parenting `status`'s prose through the JSON shape as a templating layer. Too big for this ticket; would conflate refactoring with API design. The JSON path builds its own object from the same state-file reads the prose path already does.

## Proposed approach

`scripts/hydra-status.sh` is the single owner of both outputs.

1. Add a `--json` flag in the arg-parse loop. When set, skip all color/print blocks and instead emit a single JSON object via `jq -n --argjson …` with a literal shape.
2. Add a new `scripts/hydra-ps.sh` (parallel to `hydra-status.sh`) that reads `state/active.json` and `state/repos.json`, computes `minutes_running = (now - started_at) / 60`, and emits either a human table or a JSON array. Same `--json` flag.
3. Extend the `./hydra` launcher (which is generated by `setup.sh` via heredoc — the file itself is gitignored) to add a `ps)` case to its dispatch switch. Since the launcher is regenerated by setup, the canonical change is to the heredoc in `setup.sh`. For operators whose launcher was written before this change lands, the helper scripts still work directly (`scripts/hydra-ps.sh --json`) and the next `./setup.sh` run regenerates the launcher — but setup is gated on a file-existence check, so add a migration line to the "verify / regenerate" hint inside the setup summary so operators know to `rm hydra && ./setup.sh` if they want the new subcommand wired into the launcher.
4. Publish `state/schemas/hydra-status-output.schema.json` as a Draft-07 schema. Top-level required keys: `version`, `active_workers`, `today`, `paused`, `autopickup`. Optional: `quota_health`, `pending`, `repos`. `version` is a fixed string `"1.0"` and bumps only on breaking changes — additive changes leave `version` alone and rely on `additionalProperties: true` at the object level so old schemas stay valid against new output.
5. Add self-test case `json-output-schema-valid` (kind `script`). Steps: (a) copy deterministic fixtures for `state/active.json` / `state/autopickup.json` / `state/budget-used.json` to a temp dir; (b) run `HYDRA_STATE_DIR=<tmp> scripts/hydra-status.sh --json` and `scripts/hydra-ps.sh --json`; (c) pipe each through `jq empty` (must not error); (d) if `ajv` is on PATH, validate against `state/schemas/hydra-status-output.schema.json`; (e) assert `version == "1.0"` and `.active_workers | type == "array"`.
6. README: add a 6-line "Driving Hydra from an agent" sub-section under the existing `### CLI commands (no chat required)` heading. Content: one sentence framing, then a 3-line fenced code block showing `curl` / `jq` usage.
7. `mcp/hydra-tools.json`: update the `_description` at the top to mention `state/schemas/hydra-status-output.schema.json` as the canonical shape for `hydra.get_status`'s output. No shape change — the declared output schema already matches.

### Alternatives considered

- **Alternative A — full subcommand rewrite in Python.** Rewrite `hydra-status.sh` in Python for better JSON formatting and argparse. Rejected: introduces a Python runtime dep on the hot path (today only `jq` is required); the existing bash script already uses `jq_or` helpers that do most of what's needed; and it'd break the "each `./hydra` subcommand is one small bash helper" pattern.
- **Alternative B — add JSON output to the commander's chat-level `status` command only.** Rejected: misses the point. The whole value is that `./hydra` subcommands are script-safe without launching claude, and `--json` is about making the script-safe path also machine-parseable.
- **Alternative C — reuse the MCP server (#72) as the single JSON source of truth.** Deferred, not rejected. Once the MCP binary is universally deployed, the CLI `--json` mode becomes a thin `curl http://localhost:MCP_PORT/tools/hydra.get_status` wrapper. For now, the CLI reads the same state files directly — the schemas stay aligned, so the eventual cutover is transparent to callers.

## Test plan

- **Self-test `json-output-schema-valid`** (executable today, kind `script`): see Approach step 5. Exercises both subcommands against deterministic fixtures; hard-fails if either emits malformed JSON or drifts from schema. A worker or operator who regresses the JSON shape would see this case turn red locally and on CI.
- **Manual smoke**: `./hydra status --json | jq '.version'` should print `"1.0"`; `./hydra status --json | jq '.active_workers | length'` should match the integer in the prose `Active: N worker(s)` line.
- **Fixture-driven regression**: the self-test fixtures cover empty state (no workers, autopickup off), busy state (three workers of mixed tiers, autopickup on), and paused state. Each path hits a different branch of the JSON builder.
- **CI spec-presence check**: this spec file satisfies the `scripts/spec-presence-check.sh` requirement for any PR touching non-trivial paths.
- **CI syntax-check**: the new `scripts/hydra-ps.sh` and edits to `scripts/hydra-status.sh` go through `bash -n` syntax validation, and the new schema file goes through `jq empty` validation, both via the existing `syntax-check.sh` workflow.

## Risks / rollback

- **Risk: bash JSON assembly emits invalid JSON on some shell edge cases.** `jq -n --argjson` is strict about its inputs; if a state-file field ends up as an empty string where a number is expected, the build fails and the script exits non-zero. Mitigation: the existing `jq_or` helper already produces typed fallbacks (`0` for numbers, `false` for booleans), so the JSON path uses the same fallbacks the prose path already trusts. The self-test's empty-state fixture exercises exactly this.
- **Risk: schema drifts from emitted output.** Mitigation: the self-test validates against the schema on every CI run, so drift is caught the same commit it's introduced.
- **Risk: operators with a pre-existing `./hydra` launcher don't get the new `ps` subcommand wired in.** Mitigation: document the `rm hydra && ./setup.sh` regenerate step in the README sub-section. Helper scripts still work directly, so this is a UX degradation, not a functional break.
- **Rollback**: revert the implementing commits. `state/schemas/hydra-status-output.schema.json` and the self-test case can be deleted without side effects; the `--json` flag paths are additive and guarded by explicit flag checks. Pre-existing prose callers see zero change.
