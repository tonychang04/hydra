# Runnable 10-minute Hydra demo repo

## Problem

A potential operator arrives on the Hydra README, reads the "auto-machine that clears your ticket queue while you sleep" pitch, and has no fast path to *see* that claim. The only way to get a feel for the loop today is to:

1. Install Hydra.
2. Point it at one of your own repos.
3. File a real ticket you care about.
4. Wait, and hope.

That's a multi-hour commitment before step 1 produces any evidence the system works. It also mixes "does Hydra work at all?" with "does Hydra work on *my* codebase?" — two separate failure modes that are hard to distinguish when neither has been validated.

We need a runnable, zero-install, well-scoped demo that exercises the full loop (T1 auto-merge, T2 human-merge, T3 refusal) end-to-end in roughly 10 minutes of wall-clock. It needs to be cheap to try, honest about what Hydra does (including T3 refusal), and small enough that the operator can read every file in a coffee break.

## Goals / non-goals

**Goals:**

- A `examples/hello-hydra-demo/` subtree with a minimal Node project, three fixture issue bodies, a one-command issue-filing script, and a walkthrough doc.
- Zero external deps beyond what Node 18+ ships with — no `npm install`, no lockfile, no docker, no CI config.
- All three tier outcomes are exercised in one ~10-minute session: T1 auto-merge, T2 review-gated human-merge, T3 refusal-with-commander-stuck-label.
- The fixture issue filing script is idempotent so operators can re-run without duplicating issues.
- A self-test golden case catches silent rot in the demo (broken tests, missing fixture files, stripped frontmatter, broken DEMO.md cross-references).
- A top-level README entry advertises the demo near the top — before the "The loop" section, after the glossary — so new operators see it before reading the full architecture.

**Non-goals:**

- Teaching Hydra configuration (`state/repos.json` semantics, policy tuning, memory hygiene). DEMO.md links to the relevant top-level docs; it does not re-explain them.
- Exhaustive coverage of worker types (`worker-test-discovery`, `worker-conflict-resolver`) — the demo stays on the happy path for `worker-implementation` + `worker-review`.
- Auto-filing the fixture issues at install time or on any PR merge. The filing script only fires when the operator invokes it. That's load-bearing — a demo that silently spams issues at install would be a worse first impression than no demo.
- Turning the demo into a Jest/Vitest/Mocha showcase. Node's built-in test runner is intentional — zero deps keeps the demo trustworthy.
- Supporting Node <18. The test suite uses `node:test`, which predates the v18 minimum but we set v18 as the floor to match Hydra's own Node expectations.

## Proposed approach

A new subtree at `examples/hello-hydra-demo/` with five first-class files plus `fixture-issues/` and `src/`:

```
examples/hello-hydra-demo/
├── README.md                                — minimal usage + pointer to DEMO.md. Contains one intentional typo ("hollo-hydra-demo" heading) that is the T1 target.
├── DEMO.md                                  — the 10-minute walkthrough (9 steps).
├── package.json                             — zero-dep Node manifest. "test" script runs node --test.
├── gh-filings.sh                            — one-command issue filer. Idempotent. Only fires when operator invokes it.
├── src/
│   ├── greet.js                             — greet(name) returning "Hello, <name>!" with falsy-fallback to "world".
│   └── greet.test.js                        — two passing tests under node:test.
└── fixture-issues/
    ├── 01-t1-typo.md                        — T1 target. Frontmatter + body. Fixes the hollo→hello typo.
    ├── 02-t2-farewell.md                    — T2 target. Adds farewell() mirroring greet().
    └── 03-t3-rotate-key.md                  — T3 target. References src/auth/keys.js (path shape triggers refusal).
```

The fixture issues deliberately target the three tier outcomes so a single 10-minute run exercises every branch of the merge-gate policy. #1 lives purely in docs, so CI green == node:test green, so auto-merge fires. #2 adds a new source file + tests, so it's real code work but stays within T2 additive scope. #3 references a path that matches the `*auth*` T3 tripwire in `policy.md` and names crypto material directly — commander refuses **before** any worker spawns, which is the architecturally important thing to demonstrate (refusal is cheap and happens at classification time, not at review time).

The filing script (`gh-filings.sh`) reads YAML-ish frontmatter from each fixture markdown file, extracts the `title:` field, skips issues with that exact title already on the target repo (idempotence), and opens the rest via `gh issue create --assignee @me --body-file <post-frontmatter body>`. The idempotence matters because the demo's whole value prop is "cheap to try" — if re-running the script duplicates issues, the operator pays a cleanup tax that ruins the first impression.

The self-test golden case `hello-hydra-demo-integrity` is script-kind (no worker spawn) and covers the three silent-rot modes:

1. Tests break (somebody edits greet.js and forgets to update greet.test.js).
2. Fixture frontmatter gets stripped (somebody runs a markdown formatter that eats the YAML-ish block).
3. DEMO.md loses a cross-reference to one of the fixture files (somebody reorganizes and forgets to update links).

It also asserts the README still carries the intentional `hollo-hydra-demo` typo — because if somebody "helpfully" fixes that typo in the demo-README, the T1 ticket's diff becomes empty and the walkthrough breaks.

The root README gets a short `## Try it in 10 minutes` subsection between the glossary ("Who's who") and "The loop", matching the ticket's placement requirement.

### Alternatives considered

- **Alternative A — external demo repo under `tonychang04/hello-hydra-demo`.** Cleaner mental model (demo lives where demo-issues get filed), but adds a second repo to maintain in lockstep with the main Hydra repo. First-time operators would have two clones to juggle. The `examples/hello-hydra-demo/` subtree lets us version the demo alongside the framework it demonstrates; operators who prefer the two-repo layout can copy the subtree out (DEMO.md's Variant B covers this).
- **Alternative B — make the demo a real npm package with Jest/Vitest + GitHub Actions CI.** More "realistic" for what most operators' repos look like, but adds install friction and npm-ecosystem churn (package-lock regeneration, dependency updates, Node version matrices). That friction is exactly what this demo needs to avoid. Zero-dep is load-bearing.
- **Alternative C — video walkthrough instead of a runnable demo.** Cheap to produce, but you can't grep a video. The runnable demo also doubles as a self-test substrate (the `hello-hydra-demo-integrity` golden case can't be built against a video).
- **Alternative D — exercise all four worker types, not just worker-implementation + worker-review.** Would require engineering a scenario where `worker-test-discovery` and `worker-conflict-resolver` both fire naturally in a 10-minute window. The conflict-resolver especially needs two parallel in-flight workers that collide — hard to make reliable. Punted; focus the demo on the happy path.

## Test plan

The self-test golden case `hello-hydra-demo-integrity` (added to `self-test/golden-cases.example.json`) covers the mechanical checks:

- `node --test examples/hello-hydra-demo/src/greet.test.js` exits 0.
- `bash -n examples/hello-hydra-demo/gh-filings.sh` clean.
- All three fixture-issue markdown files exist and parse a `title:` line out of the frontmatter.
- `examples/hello-hydra-demo/DEMO.md` exists and references all three fixture filenames by name.
- `examples/hello-hydra-demo/README.md` still contains the `hollo-hydra-demo` typo.
- `examples/hello-hydra-demo/package.json` is zero-dep (dependencies and devDependencies are either absent or empty).

Manual validation performed during the implementing PR (once):

- Ran `./gh-filings.sh` against a scratch fork; verified three issues created, second run was a clean noop.
- Ran the walkthrough end-to-end against a local Hydra; confirmed the expected outcome table (2 merged, 1 refused, ~10 min wall-clock) is realistic.

## Risks / rollback

- **Risk: the demo ages out of sync with real Hydra behavior.** If `policy.md` changes the T3 tripwire list, issue #3 might no longer be refused, and the walkthrough's Step 7 becomes a lie. Mitigation: the `hello-hydra-demo-integrity` golden case will catch the `hollo` typo and structural issues, but cannot catch policy-semantics drift. Owner habit: any change to `policy.md`'s T3 list should include a check of `examples/hello-hydra-demo/fixture-issues/03-t3-rotate-key.md`.
- **Risk: Node's built-in test runner changes output format in a future version and the self-test's `grep` for `duration_ms` stops matching.** Low-probability (node:test output has been stable since v18), but if it happens the self-test step breaks, not the demo itself. Fix: update the grep target.
- **Risk: operators run `gh-filings.sh` against the wrong repo.** The script requires an explicit `owner/repo` arg and refuses without it. No default. Worst case the operator files three issues on a repo they own but didn't mean to target — all fixture-issue-shaped and easy to close.
- **Rollback:** delete `examples/hello-hydra-demo/`, revert the root-README subsection, remove the golden case. The subtree is self-contained; no other code depends on it.

## Implementation notes

- The `gh-filings.sh` frontmatter parser is awk, not python, to keep the demo zero-install beyond Node. Bash + awk are guaranteed present on every macOS/Linux dev machine.
- `extract_title` trims surrounding double quotes on the title value so fixture files can use either `title: "foo"` or `title: foo`.
- The self-test step for package.json zero-dep uses `jq -e` with a combined length-check — one command, one exit code, clean PASS/FAIL without false positives on pretty-printed empty objects.
- DEMO.md explicitly documents both the variant-A (fork all of Hydra) and variant-B (standalone demo repo) paths because operators will try both and each has a subtly different UX.
- The hollo/hello typo in the demo-README is load-bearing. It's the T1 ticket's entire diff. Self-test asserts it's still present.
