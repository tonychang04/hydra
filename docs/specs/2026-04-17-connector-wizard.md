# `./hydra connect` — one-command connector wizard

## Problem

Today, wiring a connector (Linear / Slack / Upstream Supervisor MCP) is a
multi-step manual process the operator has to get right without help:

1. Install the MCP server: `claude mcp add <server>` — operator has to remember
   which servers Hydra understands.
2. Hand-author the config from a `.example.json` file. Some connectors don't
   ship one (linear had no template committed before this ticket); the operator
   guesses at shape.
3. Generate auth tokens, SHA-256 hash them, paste the hash into
   `state/connectors/mcp.json` by hand.
4. Restart the Commander session.
5. Hope the JSON is schema-valid. First validation error surfaces on the next
   autopickup tick, far from the mistake.

Commander is supposed to auto-clear toil, not ask the operator to author JSON
by hand. The operator asked: "is there a UI where I can `npm run dev` to
easily do it?" There isn't — Hydra is bash + claude — but there should be a
one-command wizard that mirrors the inbound side (`./hydra mcp register-agent`).

## Goals / non-goals

**Goals:**

- `./hydra connect <connector>` — interactive wizard for `linear`, `slack`,
  `supervisor`. Each does:
  1. Ensure MCP server is installed (for connectors that need one — linear
     does, supervisor does in http mode, slack doesn't — slack is its own
     inbound bot).
  2. Prompt for connector-specific fields with sensible defaults.
  3. Validate the result against `state/schemas/<connector>.schema.json`
     BEFORE writing.
  4. Atomic write (tmp + mv) into `state/linear.json` or
     `state/connectors/slack.json` or `state/connectors/mcp.json`.
  5. Idempotent: re-running shows the current config and offers an update
     path (no clobber, no rotation).
- `./hydra connect --list` — discover supported connectors from
  `state/connectors/*.json.example` + `state/*.example.json`, report install
  state per connector.
- `./hydra connect --status` — live health check. Parses `claude mcp list`
  output for `Needs authentication` / missing / broken states per MCP-backed
  connector. Lightweight `hydra doctor`-for-connectors.
- `./hydra connect --dry-run <connector>` — print the JSON it would write,
  do not write.
- JSON schemas for every connector config we write:
  `state/schemas/linear.schema.json`, `state/schemas/slack.schema.json`,
  `state/schemas/mcp.schema.json` (currently the closest thing is an implicit
  description in the `.example.json` — finish it).
- Session greeting gets a one-line connector-status tag when anything is
  broken or unauthenticated (e.g.
  `Connectors: linear broken (auth expired), slack OK`). No inline
  procedure in `CLAUDE.md` — link to this spec.
- `README.md` gets a 3–5 line "Connectors" section pointing at
  `./hydra connect`.

**Non-goals:**

- Not a browser UI. Terminal-first. No `npm run dev`. No Electron. No
  Flask. Bash + jq.
- Not a credential store. Tokens stay in the operator's env / keychain /
  `claude mcp` config. Only SHA-256 hashes land in
  `state/connectors/mcp.json` (existing pattern from
  `hydra mcp register-agent`).
- Not a rotator. This is initial wire-up only. Token rotation for the
  OUTBOUND direction (supervisor auth) is a follow-up. Rotation of INBOUND
  agent tokens already exists via `./hydra mcp register-agent <id> --rotate`.
- Not a MCP server installer wrapper beyond `claude mcp add` — no package
  manager logic, no OS-specific installers. If `claude mcp add linear` fails,
  the wizard surfaces the error and stops; the operator fixes their
  `claude mcp` environment and re-runs.
- Not mutating `CLAUDE.md`, `policy.md`, `budget.json`,
  `.claude/settings.local.json`, `.claude/agents/*.md`, or any worker
  subagent. Same hard rules as all other `./hydra` subcommands.

## Proposed approach

One entry script, three sub-functions, three schemas.

### Entry script: `scripts/hydra-connect.sh`

Dispatched from `./hydra connect …` via the existing subcommand table in
`./hydra` (generated by `setup.sh`). Same pattern as `./hydra doctor`,
`./hydra add-repo`, `./hydra mcp register-agent`.

Flags:

```
./hydra connect <connector>       # interactive wizard for one connector
./hydra connect --list            # discover + report install state of all
./hydra connect --status          # live MCP health check across all connectors
./hydra connect --dry-run <c>     # preview JSON, don't write
./hydra connect --non-interactive # never prompt; fail if a required field is missing
./hydra connect -h, --help        # usage
```

Supported `<connector>` values: `linear`, `slack`, `supervisor`.
Adding a fourth is a one-file change: add a `connect_<name>` bash function
and a schema; add the name to the dispatch `case` and the `--list` discovery
set.

### Sub-functions (all live in `hydra-connect.sh`)

**`connect_linear`:**

- Target file: `state/linear.json`
- Schema: `state/schemas/linear.schema.json` (NEW — authored in this PR)
- Bootstrap: if `state/linear.json` missing, copy from
  `state/linear.example.json` and then strip the `_description`-prefix fields
  (same pattern as `hydra mcp register-agent`'s mcp.json bootstrap).
- Preflight: check `claude mcp list 2>/dev/null | grep -q '^linear '`.
  - If not installed: print actionable next step
    (`claude mcp add linear`) and, unless `--non-interactive`, offer to
    run it for the operator. The operator still does the oauth flow themselves
    — `claude mcp add linear` is interactive the first time.
- Prompts:
  - `team_id` — free-form string, required
  - `team_name` — free-form string, default `Engineering`
  - `trigger_type` — one of `assignee | state | label`, default `assignee`
  - `trigger_value` — free-form string, default `me`
  - `repo_mapping` — one line per `<owner>/<repo>` (blank to finish); maps
    Linear issues to GitHub repos
- Write: `jq -n` build + atomic swap.
- Re-run behavior: read existing `state/linear.json`, show current values in
  prompt defaults, let operator overwrite by typing a new value or keep by
  pressing enter. No clobber of unrelated keys (merge semantics).

**`connect_slack`:**

- Target file: `state/connectors/slack.json`
- Schema: `state/schemas/slack.schema.json` (NEW)
- No MCP server — Slack is a Socket-Mode bot (`bin/hydra-slack-bot`). No
  `claude mcp add` step. Operator provides `bot_token_env` + `app_token_env`
  names (not the tokens themselves — the tokens live in Fly secrets / env).
- Prompts:
  - `enabled` — default `false` (operator flips to `true` after tokens are
    set in their environment)
  - `bot_token_env` — default `HYDRA_SLACK_BOT_TOKEN`
  - `app_token_env` — default `HYDRA_SLACK_APP_TOKEN`
  - `allowed_user_id` — required, must match `^U[A-Z0-9]+$`
  - `default_channel` — required, must match `^[CDG][A-Z0-9]+$`
  - `notification_poll_interval_sec` — default `60`, minimum `30`
  - `repos` — one line per `<owner>/<repo>` (blank to finish)
- Re-run shows current config, preserves unspecified keys.

**`connect_supervisor`:**

- Target file: `state/connectors/mcp.json` (section: `upstream_supervisor`)
- Schema: `state/schemas/mcp.schema.json` (NEW — only the fields Hydra reads
  today; `additionalProperties: true` so hand-added keys survive)
- Bootstrap: if file missing, copy from example as with mcp-register-agent
  (same code path — factor a shared helper if the duplication is obvious,
  otherwise inline and note in the comment).
- Prompts:
  - `enabled` — default `false`
  - `name` — free-form
  - `transport` — `stdio` | `http`, default `stdio`
  - `connect_url` — required — stdio command OR https URL
  - `auth_token_env` — default `HYDRA_SUPERVISOR_TOKEN`
  - `scope_granted` — hardcoded to `["resolve_question"]` (CLAUDE.md invariant)
  - `timeout_sec` — default `300`
- Writes ONLY the `upstream_supervisor` subtree; leaves `server`,
  `authorized_agents`, `audit` untouched.

### `--list`

Discovers supported connectors by scanning `state/*.example.json` +
`state/connectors/*.example.json` for the canonical names we understand
(`linear`, `slack`, `mcp` → supervisor). For each, reports:

```
linear     installed  (state/linear.json exists, schema OK)
slack      missing    (no state/connectors/slack.json)
supervisor disabled   (upstream_supervisor.enabled=false)
```

Printed as plain text columns. No `--json` in this ticket (follow-up if
asked). Exit 0 regardless — `--list` is informational, not a check.

### `--status`

Runs `claude mcp list 2>/dev/null`, parses each line for `<name>` followed
by `Needs authentication` or `✓` or an explicit error. Cross-references
with local config state (enabled / disabled). For non-MCP connectors
(slack), reports based on config presence + `enabled` flag only.

Output shape:

```
linear     OK         (mcp-installed, authenticated, state/linear.json valid)
slack      DISABLED   (state/connectors/slack.json:enabled=false)
supervisor BROKEN     (state/connectors/mcp.json:upstream_supervisor.enabled=true but connect_url missing)
```

Exit codes: `0` all OK or DISABLED; `1` at least one BROKEN. This makes
`--status` safe to call from `hydra doctor` (an enhancement — not wired in
this ticket).

### Schemas

New files:

- `state/schemas/linear.schema.json` — mirrors `state/linear.example.json`
  shape. Required: `enabled`, `teams`, `state_transitions`.
- `state/schemas/slack.schema.json` — mirrors
  `state/connectors/slack.json.example`. Required: `enabled`,
  `bot_token_env`, `app_token_env`, `allowed_user_id`, `default_channel`,
  `notification_poll_interval_sec`, `repos`, `seen_prs_state_path`.
- `state/schemas/mcp.schema.json` — mirrors
  `state/connectors/mcp.json.example`. Required: `server`,
  `authorized_agents`, `upstream_supervisor`, `audit`.
  `additionalProperties: true` everywhere so the `_description`-prefixed
  fields in the `.example.json` don't blow up validation.

Existing pattern: `state/schemas/repos.schema.json` uses Draft-07 +
`additionalProperties: true` — follow that. Same `$schema`, `$id`, `title`,
`description` layout.

### Validation wiring

- Wizard writes go through `scripts/validate-state.sh` BEFORE the atomic
  swap. If validation fails, print the ajv error (or the minimal-mode missing
  keys), delete the tmpfile, exit 1.
- The existing `scripts/validate-state.sh` derives schema from filename:
  `state/linear.json` → `state/schemas/linear.schema.json`. That already
  works for `linear.json`; we need to either (a) rename
  `state/connectors/slack.json` to live at `state/slack.json` (NO — breaks
  existing bot wiring), or (b) pass `--schema` explicitly in the connect
  script. We go with (b) — explicit `--schema <path>` call.
- `scripts/validate-state-all.sh` already iterates state files and picks up
  anything under `state/*.json`. For files under `state/connectors/` it
  won't find the schema automatically; extending validate-state-all is
  noted as a follow-up but NOT done in this ticket (out of scope: bloat-
  adjacent; #138 is already refactoring preflight).

### Session greeting

One-line addition to the existing greeting in `CLAUDE.md`:

```
Connectors: linear OK, slack DISABLED, supervisor not-configured.
```

Computed by calling `scripts/hydra-connect.sh --status --quiet` (new flag
— prints just the one-line summary). Only shown when at least one
connector is BROKEN or MISCONFIGURED. Otherwise suppressed (silence is a
feature).

**No inline procedure in CLAUDE.md.** One-liner + link to this spec.
(`memory/feedback_claudemd_bloat.md` + ticket #138 dictate the shape of
this addition.)

Add under the operator-facing subcommand section:

    ./hydra connect linear

    ./hydra connect --status

    ./hydra connect --dry-run linear

## Risks / rollback

- **`claude mcp add` prompts interactively.** We don't block on it — the
  script exec's `claude mcp add linear` and returns the child's exit code.
  If the operator aborts mid-oauth, nothing on the Hydra side is written
  (we only write AFTER the prompts, and prompts run after `claude mcp add`
  succeeds). No partial state.
- **Schema drift** if the example `.json` file evolves but the schema doesn't.
  Mitigation: schemas intentionally use `additionalProperties: true` for the
  top level + nested objects (mirrors existing `repos.schema.json` pattern)
  so added documentation fields don't break validation. Required-keys list
  stays narrow — only fields Hydra actually reads.
- **Re-run on a config that's already valid** must not clobber unrelated
  fields. We merge: read existing → jq merge with prompted values → write.
  Unit-tested via the `--dry-run` path reading an existing fixture.
- **Rollback:** deleting `scripts/hydra-connect.sh` and the three new
  schemas restores prior behavior — no other script depends on them. The
  `./hydra connect` subcommand branch is additive in the generated launcher;
  removing it doesn't affect existing subcommands.
