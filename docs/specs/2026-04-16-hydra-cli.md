# hydra CLI subcommands — add-repo / status / pause / issue without chat

## Problem

Today every operator touch-point requires opening the Commander chat session (`./hydra`). Even routine operations — checking what's running, pausing the machine, adding a new repo to `state/repos.json`, queueing a specific GitHub issue — pulls a human into a conversation. That's fine for novel decisions, hostile for muscle-memory ops:

- Operator sees a burning PR in Slack and wants to pause; has to launch the Commander, wait for the greeting, type `pause`, wait for the confirm.
- Operator onboards a fourth repo; has to hand-edit `state/repos.json` with an external editor (no schema help, no duplicate check), restart the Commander session so the new entry gets picked up.
- Operator wants a one-shot queue of a GitHub issue they just filed, without opening chat — no way to do that today.
- Shell scripts / cron / CI can't `./hydra pause` — the launcher exec's claude, full-stop.

Concrete signal: ticket #25. The doctor pattern from #36 already established the shape (sibling subcommand that dispatches to a helper script). This spec generalizes that shape to the full daily-ops vocabulary.

## Goals / non-goals

**Goals:**
- `./hydra add-repo <owner>/<name>` — interactive wizard, validates + atomic-writes `state/repos.json`
- `./hydra remove-repo <owner>/<name>` — remove entry, preserve JSON formatting
- `./hydra list-repos` — table of configured repos (name / trigger / enabled / last poll)
- `./hydra status` — read-only snapshot of what the Commander's session-start greeting would show (ticket counts, active workers, throughput, autopickup state) without launching a session
- `./hydra pause` / `./hydra resume` — toggle the `PAUSE` file, confirm current state
- `./hydra issue <url-or-owner/repo/num>` — append to `state/pending-dispatches.json` so the next autopickup tick drains it first
- All subcommands runnable from any terminal (no Commander chat) — operator can chain them from shell scripts or cron
- Each helper script is a standalone `set -euo pipefail` bash + jq program with `--help` / `usage()`
- Atomic writes: always tmp + mv, never half-write state/repos.json
- Validate against `state/schemas/repos.schema.json` when the schema file exists; skip gracefully if not
- Self-test golden case `hydra-cli-subcommands-valid` (kind: script) — syntax valid, each `--help` works, list-repos against fixture produces expected shape
- Launcher wiring mirrors the doctor pattern: subcommand branch in `setup.sh`'s generated `./hydra` template
- README.md + USING.md updates so operators discover the commands

**Non-goals:**
- **These subcommands do NOT spawn workers or open PRs.** They manage Hydra's own state. `./hydra issue` is the one exception — it queues, it does not spawn.
- Not a replacement for Commander chat. Novel decisions, tier classification on gray-area tickets, and conflict resolution all still go through chat. These commands are for routine plumbing.
- No interactive TUI, no curses. Interactive wizard = line-at-a-time prompts, readable when piped.
- No cloud / headless mode. All subcommands operate on the local worktree's state files. Phase 2's MCP interface is a different transport (see `docs/mcp-tool-contract.md`).
- Doesn't touch the permission denylist, `policy.md`, `budget.json`, worker subagents, or `CLAUDE.md`. Same hard rules as doctor.
- Not a settings editor. Enable / disable toggles, trigger changes, merge-policy overrides are edited by hand in `state/repos.json` — add-repo is only for the initial create; disable is `--force` path on add-repo or manual edit.

## Proposed approach

Seven helper scripts under `scripts/`, each a leaf `.sh` file matching the doctor pattern:

1. **`scripts/hydra-add-repo.sh`** — wizard. Parses `owner/name`, prompts (with auto-suggested default from `git remote get-url origin` lookup) for `local_path`, `ticket_trigger`, `enabled`. Reads `state/repos.json` (creates from example if missing), validates the owner/name format, validates local_path exists + is a git repo, refuses duplicates (`--force` to override), appends the new entry with jq, atomic-writes via tmp + mv. Prints a formatted summary. Validates the result against `state/schemas/repos.schema.json` if a validator is available; warns otherwise.
2. **`scripts/hydra-remove-repo.sh`** — takes `owner/name`, confirms it exists, removes the entry via jq, atomic-writes. Noop-with-message if not present.
3. **`scripts/hydra-list-repos.sh`** — reads `state/repos.json`, formats a text table. Joins in `state/autopickup.json:last_run` (and per-repo last-poll if we ever track it — right now that's a single global timestamp, so we show it once at the top).
4. **`scripts/hydra-status.sh`** — read-only. Replicates the Commander session-greeting line: count of assignee-triggered open issues across enabled repos (best-effort via `gh issue list --assignee @me --state open`, skips if `gh` unavailable), active worker count from `state/active.json`, today's throughput from `state/budget-used.json`, autopickup state from `state/autopickup.json`, pause file status. Does NOT need a session.
5. **`scripts/hydra-pause.sh`** — creates `PAUSE` file at repo root (idempotent), prints current state.
6. **`scripts/hydra-resume.sh`** — removes `PAUSE` file if present (idempotent), prints current state.
7. **`scripts/hydra-issue.sh`** — parses arg as a URL, `owner/repo#N`, or `owner/repo/N`. Normalizes to `{owner, name, number}`. Appends to `state/pending-dispatches.json` (created if missing, gitignored). Next autopickup tick drains this file first, BEFORE the usual trigger query. Does NOT spawn a worker.

**Launcher wiring** in `setup.sh`'s generated `./hydra` template. Current code has a single `if [[ "${1:-}" == "doctor" ]]` branch; expand to a `case` dispatch that routes each known subcommand, preserving `--version` short-circuit and the existing launch path.

**Atomic write pattern** (used by all write paths):
```bash
tmp="$(mktemp)"
jq '...' "$state_file" > "$tmp"
mv "$tmp" "$state_file"
```
Mv is atomic on the same filesystem, so a concurrent read never sees a half-written file.

**State file conventions:**
- `state/pending-dispatches.json` is new (gitignored — same class as `state/active.json`). Shape: `{"queue": [{"owner", "name", "number", "added_at", "source"}]}`. Commander's autopickup tick reads and truncates on drain.
- `PAUSE` file at repo root (already documented in `CLAUDE.md`). Presence = paused; absence = not paused. Content is informational only.

### Alternatives considered

- **Alternative A: Expose these as Commander chat commands only.** Rejected — the whole point of #25 is to make daily-ops muscle-memory *without* opening chat. Shell scripts, cron, CI runners, phone-via-SSH — none of those want to spawn a Claude session just to `./hydra pause`.
- **Alternative B: One monolithic `./hydra admin <subcommand>` dispatcher.** Rejected — extra typing, less discoverable. `./hydra status` reads the same as `git status`. The doctor precedent is sibling subcommands, and that pattern is already familiar.
- **Alternative C: Python CLI with argparse + jsonschema.** Rejected — Hydra's hard rule is bash + jq only on the installable surface. Adding a Python dep for CLI polish isn't a good trade.
- **Alternative D: Edit `state/repos.json` via the Commander subagent (spawn a worker-edit or use Edit tool).** Rejected — subagent spawn is heavy for a config edit, and the Commander has an explicit deny on editing its own `CLAUDE.md`/`policy.md`. Direct shell edit (with schema validation) is simpler and matches the file's ownership model — the operator owns `state/repos.json`.

## Test plan

- `bash -n scripts/hydra-*.sh` — syntax-clean on all new scripts.
- Each new script, run with `--help`, exits 0 and prints usage banner.
- `scripts/hydra-add-repo.sh foo/bar --local-path /nonexistent` — exits non-zero with a clear "path does not exist" error.
- `scripts/hydra-add-repo.sh foo/bar --local-path <valid-repo> --non-interactive --trigger assignee` — appends entry, returns 0.
- Running add-repo again with the same `owner/name` — exits 2 with duplicate-entry error. Adding `--force` makes it succeed and replace.
- `scripts/hydra-pause.sh` — creates PAUSE; `scripts/hydra-resume.sh` — removes it. Idempotent on repeat calls.
- `scripts/hydra-issue.sh https://github.com/foo/bar/issues/42` — appends to `state/pending-dispatches.json`; second call with same URL doesn't duplicate (best-effort dedup on owner/name/number).
- `scripts/hydra-list-repos.sh` against the fixture repos.json in `self-test/fixtures/hydra-cli/` produces a table with 3 rows (matching the fixture).
- Self-test golden case `hydra-cli-subcommands-valid` (kind: script): syntax check, `--help` for each, list-repos fixture assertion, pause/resume roundtrip.
- `jq . self-test/golden-cases.example.json` — still parses cleanly after the new case is added.

## Risks / rollback

- **Risk: A concurrent Commander session + `./hydra add-repo` race on `state/repos.json`.** Acceptable — atomic mv ensures no half-write. The worst case is the Commander reads the pre-add version on its next pickup tick, then catches the new entry the tick after. We already assume `state/repos.json` can change mid-session.
- **Risk: `./hydra issue` queues a T3 ticket; next autopickup tick tries to classify it anyway.** Fine — the Commander's classification logic stops at T3 and refuses as normal. The queue is a *pickup hint*, not a bypass of the tier gate.
- **Risk: A badly-formed `state/pending-dispatches.json` crashes an autopickup tick.** Mitigation: `hydra-issue.sh` schema-validates its write; Commander's tick reads defensively (`jq -r '.queue // []'` with fallback). If a tick's queue read fails, log and skip the queue (fall back to the normal trigger path).
- **Risk: The interactive wizard fights non-interactive invocation from a script.** Mitigation: every prompt has a `--flag` variant; `--non-interactive` mode requires all flags or errors out cleanly.
- **Rollback:** delete the seven new scripts, revert the `setup.sh` launcher template patch, revert the README / USING.md / self-test additions. No persistent state is migrated; `state/pending-dispatches.json` is transient and gitignored. Clean revert.

## Implementation notes

- `scripts/hydra-status.sh` deliberately does NOT hit the network unless `gh` is present AND `--with-tickets` is passed — a pure read of local state is the fast path; the ticket count is a best-effort enhancement.
- `scripts/hydra-issue.sh` treats `state/pending-dispatches.json` as append-only within a run; Commander owns the drain + truncate. This separates producer (operator/CLI) from consumer (autopickup tick) cleanly.
- The "auto-suggest `local_path`" in add-repo searches `~/`, `~/projects/`, `~/code/`, `~/work/`, `~/src/`, `~/git/`, and `$PWD` for a directory whose `git remote get-url origin` matches the requested owner/name. If nothing matches, the prompt falls back to a blank default. Keep this lookup fast (one-deep only) — don't descend into arbitrary subdirs.
- Every script sources `.hydra.env` if present so `HYDRA_ROOT` / `HYDRA_EXTERNAL_MEMORY_DIR` are available — matches the doctor pattern. When `.hydra.env` is absent, the script resolves paths relative to its own location and proceeds.
- Don't add checks that require editing `.claude/settings.local.json` — these tools run outside the permission sandbox (they're invoked by the human, not by a Claude agent). If a future feature wants them callable by Commander, add the Bash permission entry at that point.
