# CI on every PR

## Problem

Hydra has accumulated a web of shell scripts (`setup.sh`, `scripts/*.sh`, `self-test/*.sh`, fixtures), JSON config (`budget.json`, `state/*.example.json`, `state/schemas/*.json`, `mcp/hydra-tools.json`), and markdown specs (`docs/specs/*.md`) — with **no CI**. A worker-implementation (or a human contributor) can push a PR with a broken `bash -n`, malformed JSON, a missing spec frontmatter, or half-resolved conflict markers and nothing catches it until somebody pulls main and trips over it.

At the same time, `DEVELOPING.md` declares spec-driven development "non-negotiable" for changes to framework files (`CLAUDE.md`, `policy.md`, `budget.json`, `.claude/agents/*.md`, the settings template) — but the rule is enforced by hope, not by a gate. A worker can skip the spec and it'll merge.

Issue #15 also landed a real `self-test/run.sh` script-kind runner. It works locally. Nothing exercises it on CI, so a prompt edit in `CLAUDE.md` or a regex tweak in `scripts/validate-learning-entry.sh` can silently break the harness between merges.

All three gaps have the same root cause: no `.github/workflows/` directory.

## Goals / non-goals

**Goals:**

- Run a fast (<2 min wall-clock) CI on every PR and every push to `main`.
- Fail PRs that ship broken shell syntax, malformed JSON, specs without required frontmatter, or leftover merge conflict markers.
- Enforce the spec-presence rule from `DEVELOPING.md`: framework-file changes MUST include a `docs/specs/YYYY-MM-DD-*.md` in the same PR. Escape hatch: the label `trivial` bypasses the check.
- Run `./self-test/run.sh --kind=script` on every PR so prompt/script regressions fail loudly.
- Work with only the default `GITHUB_TOKEN` — no extra secrets.
- Readable CI status badge in `README.md`.

**Non-goals:**

- Running the `worker-implementation`, `worker-subagent`, `commander-inline`, or `commander-self` kinds. These need Claude auth / real commander orchestration (ticket #15 documents this). They SKIP in the script runner, which is the correct behavior.
- Docker-heavy end-to-end cases. The `requires_docker` cases (e.g. the localstack-backed `s3-backend-roundtrip`) still run locally where docker is present, but need to SKIP cleanly when docker is unavailable so CI stays fast and green. This spec adds a minimal `requires` check to `self-test/run.sh` for that purpose.
- Deploy, release, or merge-queue logic. CI is a gate, not a pipeline.
- Linting docstrings, formatting, or style beyond `bash -n` + `jq empty` + frontmatter presence. Style is a later ticket.

## Design

Four parallel jobs in a single workflow. Each runs on its own `ubuntu-latest` runner with a short `timeout-minutes`. All scripts `set -euo pipefail`.

### 1. `syntax`

Composite check run via `.github/workflows/scripts/syntax-check.sh`:

- **bash -n** on `setup.sh`, `install.sh`, `scripts/*.sh`, `self-test/*.sh`, `self-test/fixtures/**/*.sh`, `infra/*.sh` (if present), `.github/workflows/scripts/*.sh`.
- **jq empty** on every known JSON target: `state/*.example.json`, `state/schemas/*.json`, `self-test/golden-cases.example.json`, `mcp/hydra-tools.json`, `docs/specs/*.json` (none today, future-proof), `budget.json`.
- **Frontmatter check** on `docs/specs/*.md` (excluding `README.md` and `TEMPLATE.md`): the first line MUST be `---`, and the frontmatter block MUST contain `status:` and `date:` with non-empty values.

### 2. `spec-presence`

Runs only on `pull_request` events. Reads `git diff --name-only origin/$BASE_REF...HEAD` to enumerate changed files, compares against the framework-file list, and fails if any framework file was touched without a companion `docs/specs/YYYY-MM-DD-*.md` modification or addition.

**Framework-file list (authoritative — mirrors `DEVELOPING.md`):**

- `CLAUDE.md`
- `policy.md`
- `budget.json`
- `.claude/agents/*.md`
- `.claude/settings.local.json.example`

**Escape hatch:** `gh pr view $PR_NUMBER --json labels` — if the PR has the label `trivial`, skip the check. This exists for typo fixes, comment tweaks, whitespace-only patches where a spec would be noise.

On failure, the error message is:

> Framework-file change without a spec. Add docs/specs/\<YYYY-MM-DD\>-\<slug\>.md per DEVELOPING.md.

### 3. `self-test`

Runs `./self-test/run.sh --kind=script`. The runner already SKIPs `worker-implementation`, `worker-subagent`, `commander-inline`, `commander-self` per ticket #15. This spec adds a small enhancement: `self-test/run.sh` now honors a `requires` array on each case (`requires_docker`, `requires_aws`) and SKIPs cases whose prerequisites aren't met on the runner. Unknown `requires` tokens are treated as satisfied (forward-compat).

### 4. `no-conflict-markers`

`grep -rnE '^(<{7}|={7}|>{7}) ' .` with `.git` excluded. The leading `^` and trailing space ensure we only match real markers (start-of-line followed by a space), not prose that happens to mention the word "conflict".

## Gated vs advisory

Everything in this CI is **gated**. Failing any job blocks the PR. No advisory-only checks today — if we want something to be advisory later, we can mark it `continue-on-error: true`.

## Meta-note

This PR is the first one that will run its own CI. The only way to know the `spec-presence` gate actually works is for the PR itself to satisfy it — i.e., this spec must be present and the workflow must see it in `git diff`. If CI fails on the `spec-presence` job for this PR, that's self-referential proof the gate works.

## Test plan

1. **Local bash -n** on the two new scripts — OK.
2. **Local `syntax-check.sh` pass** — OK against current tree.
3. **Local `./self-test/run.sh --kind=script`** — all cases pass or SKIP cleanly (no FAILs).
4. **Workflow YAML valid** — loaded by GitHub Actions after PR opens.
5. **Meta-test** — the CI run on this PR is the test of the CI itself. If it goes green, the gate passed its own enforcement. If `spec-presence` fails on this PR, the gate demonstrably works.
