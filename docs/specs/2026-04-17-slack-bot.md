# Slack bot: DM-based chat surface for cloud commander

Implements: ticket [#74](https://github.com/tonychang04/hydra/issues/74) (T2).

## Problem

Once Commander is in agent-only / cloud-hosted mode (see `docs/specs/2026-04-16-mcp-agent-interface.md`, `docs/specs/2026-04-16-vps-deployment.md`), the operator has no phone-friendly surface for either direction of traffic:

- **Inbound** (operator → Commander): today, issuing a command like `pick up 2` requires `fly ssh console` + `tmux attach`, or — once ticket #72 lands — a Claude Code session on a laptop calling `hydra.*` MCP tools. Neither is realistic when the operator is away from their desk.
- **Outbound** (Commander → operator): when a PR passes `worker-review` and is ready to merge (T2+ requires human attestation per `policy.md`), the operator has no push-style notification. Things pile up until the next `fly logs` session.

The `docs/phase2-vps-runbook.md` "Setting up Slack" link already promises this feature (marked "follow-up #42" in the old runbook text) but no GitHub issue exists. The `docs/specs/2026-04-16-mcp-agent-interface.md` spec explicitly supersedes the old #42 (and #49 Discord) because per-platform chat bots are the wrong primary surface — MCP is. That framing still holds: Slack is **not** a replacement for MCP. It is a thin Slack-DM ↔ Commander bridge that:

1. Leans on MCP once #72 lands (the bot becomes just another authorized MCP client, with its own bearer token).
2. Falls back to direct state-file reads until #72 lands, so Phase 1 operators still get value.

The supervisor-agent abstraction from the MCP interface spec stays intact: the Slack bot is one possible supervisor transport, not a first-class Commander concept.

## Goals

- **G1 — Single-operator DM surface.** Bot listens for DMs from exactly one allowlisted Slack user ID. No channels, no multi-user, no broadcast.
- **G2 — Two commands hard-wired.** `status` (returns current Commander snapshot) and `pickup [N]` (triggers pickup of N tickets, default 1). Every other verb is a `NotImplemented` stub with a polite "coming in follow-up" reply.
- **G3 — One proactive notification trigger.** When any PR receives the `commander-review-clean` label (applied by Commander after `worker-review` clears it), the bot DMs a one-line notice with the PR link.
- **G4 — Refuse-to-start safety.** Missing config, `enabled: false`, or missing token env vars → process exits 1 with a clear log message. Never boot a half-configured bot.
- **G5 — Audit log.** Every DM (inbound) and response (outbound) appends one JSON line to `logs/slack-audit.jsonl`. No raw message bodies (may contain tokens) — hash the text, log envelope only.
- **G6 — Clean swap point.** Every place the bot reads Commander state directly (not via MCP) is marked with a `# PHASE1-FALLBACK` inline comment. After #72 lands the swap is mechanical: replace each fallback with a call to `mcp_client.call('hydra.<tool>')`.

## Non-goals

- **Multi-user / channel / broadcast.** Single-operator DM only. Enforced by the `allowed_user_id` check.
- **Block Kit / rich UI / modals / buttons.** Plain markdown replies.
- **Other chat platforms.** Discord / Teams / SMS are each their own ticket.
- **Custom slash commands (`/hydra-status`).** DM text parsing only. No workspace-level slash-command registration.
- **`merge` / `reject` / `retry` command handlers.** Scaffolded as `NotImplemented` stubs so the pattern is committed but the semantics are a follow-up.
- **Webhook-driven notifications.** Phase 1 polls `commander-review-clean` PRs every 60s. Webhook-driven is a follow-up once there's a webhook receiver (issue #40).
- **Editing `infra/entrypoint.sh`, `infra/fly.toml`, or `CLAUDE.md`.** Ticket #72 is touching entrypoint + fly.toml; CLAUDE.md is crowded. All three get a post-landing integration checklist in the PR body instead.
- **Launching the bot from `entrypoint.sh`.** The bot ships as a standalone launcher (`bin/hydra-slack-bot`). Wiring it into `entrypoint.sh` is a #72 follow-up; the runbook documents the one-line change needed there.

## Proposed approach

### Library: slack-bolt (Python, socket mode)

`slack-bolt` is Slack's official Python SDK for bot apps. Socket mode means the bot opens an outbound WebSocket to Slack — no public HTTP endpoint on the Fly Machine, no URL to register in the Slack app settings, no firewall holes. Exactly right for a Fly `auto_stop = suspend` Machine.

Two env-var tokens are required:

- `bot_token_env` → the Slack app's bot user OAuth token (`xoxb-…`). Used to post messages.
- `app_token_env` → the Slack app-level token (`xapp-…`), scoped `connections:write`. Used to open the socket-mode WebSocket.

Both are loaded at startup via the env-var-name pointers in `state/connectors/slack.json` — never inlined in the config file.

### Config: `state/connectors/slack.json.example`

```jsonc
{
  "_description": "Copy to state/connectors/slack.json and fill in. Controls the Hydra Slack bot (bin/hydra-slack-bot). Bot runs in the Commander process (cloud or local), listens for DMs from exactly one allowlisted user, and posts proactive notifications on state transitions. Gitignored under state/connectors/*.json — the .example mirror is the only committed variant.",
  "enabled": false,
  "_enabled_note": "Set to true to let bin/hydra-slack-bot start. Default false so a missing config is inert.",
  "bot_token_env": "HYDRA_SLACK_BOT_TOKEN",
  "_bot_token_env_note": "Name of the env var holding the xoxb- bot user OAuth token. Do NOT paste the token into this file.",
  "app_token_env": "HYDRA_SLACK_APP_TOKEN",
  "_app_token_env_note": "Name of the env var holding the xapp- app-level token (scoped connections:write) for socket mode.",
  "allowed_user_id": "U0000000000",
  "_allowed_user_id_note": "Exactly ONE Slack user ID (not display name). DMs from any other user are ignored with a single audit-log line. Find yours via Slack profile → three-dot menu → Copy member ID.",
  "default_channel": "D0000000000",
  "_default_channel_note": "Fallback channel (DM conversation ID) for proactive notifications if DMing allowed_user_id fails (user removed the bot from their workspace, etc.). Usually the same DM conversation ID as the one between allowed_user_id and the bot.",
  "notification_poll_interval_sec": 60,
  "_notification_poll_interval_sec_note": "How often to poll for newly-labeled commander-review-clean PRs. Default 60s. Min 30s (any tighter hits GitHub rate limits).",
  "repos": [
    "tonychang04/hydra"
  ],
  "_repos_note": "Which repos to poll for the commander-review-clean label. Phase 1: must be listed explicitly. Phase 2: should auto-derive from state/repos.json once #72 lands."
}
```

**Why `_env` suffix convention:** matches `state/connectors/mcp.json.example:upstream_supervisor.auth_token_env`. Consistent with how Commander already handles secrets — the config names an env var, the operator sets the env var, the config file stays safe to commit into a password manager or backup.

### Launcher: `bin/hydra-slack-bot`

Single Python script (~300 LOC). Responsibilities:

1. **Startup refusal loop.** Read `state/connectors/slack.json`. Refuse to start if:
   - File is missing (exit 1, log: "slack bot config not found at state/connectors/slack.json")
   - `enabled: false` (exit 0, log: "slack bot disabled in config; exiting cleanly")
   - Required env vars (named by `bot_token_env` / `app_token_env`) are unset (exit 1, log: "env var $NAME unset")
2. **Message handler (inbound).** On any DM, check `event.user == allowed_user_id`. If no, log one audit line (`action: refused-unauthorized-user`) and silently return. If yes, parse first token as command:
   - `status` → return Commander snapshot
   - `pickup` or `pickup N` → return pickup result
   - `merge`, `reject`, `retry`, `pause`, `resume` → polite "coming in follow-up" reply, log `action: not-implemented`
   - Anything else → "unrecognized command. supported: `status`, `pickup [N]`"
3. **Notification loop (outbound).** Background thread, polls every `notification_poll_interval_sec`. For each repo in config, runs `gh pr list --repo <repo> --label commander-review-clean --state open --json number,title,labels,url`. For each PR not seen before (tracked in `state/slack-notified-prs.json`), DM the operator and append to the seen-set. The seen-set persists so a bot restart doesn't re-notify.
4. **Audit logging.** Every DM-in and DM-out appends to `logs/slack-audit.jsonl` — `{ts, direction, user_id, text_sha256, action, worker_ids?}`. `text_sha256` is the first 16 hex chars of SHA-256 of the raw text, so two identical messages deduplicate in post-hoc analysis without ever logging raw content.
5. **Graceful shutdown.** `SIGTERM` / `SIGINT` closes the WebSocket, flushes the audit log, exits 0. Important for `fly deploy` rolling restarts — unclean shutdowns leak WebSocket connections on Slack's side and cause duplicate-delivery on reconnect.

### PHASE1-FALLBACK pattern

Phase 1 (today, pre-#72) the bot reads Commander state directly:

- `status` handler reads `state/active.json`, `state/budget-used.json`, `state/autopickup.json`.
- `pickup N` handler writes a dispatch request to `state/pending-dispatches.json` and polls for the Commander-side response. This mirrors the existing file-based dispatch pattern that `./hydra` uses from terminal mode. (If that file doesn't exist yet — e.g. older Commander versions — the handler replies "pickup not wired in Phase 1 yet; use `fly ssh attach` instead" and logs the gap.)

Each such read/write is tagged with an inline `# PHASE1-FALLBACK: swap to mcp_client.call('hydra.<tool>') after #72 lands` comment, making the future refactor mechanical — grep for `PHASE1-FALLBACK`, replace one call at a time.

Phase 2 (after #72): every fallback site becomes one MCP call, with the bot's bearer token registered in `state/connectors/mcp.json:authorized_agents[]` with scope `["read", "spawn"]` (no `merge`, no `admin` — Slack only does observation + pickup in this ticket's scope).

### Audit log shape

`logs/slack-audit.jsonl` is append-only JSONL. One line per event:

```jsonc
// inbound DM
{"ts": "2026-04-17T10:00:00Z", "direction": "in", "user_id": "U...", "text_sha256": "abc123...", "action": "command-status", "parsed_cmd": "status"}

// outbound reply
{"ts": "2026-04-17T10:00:01Z", "direction": "out", "user_id": "U...", "text_sha256": "def456...", "action": "reply-status"}

// unauthorized
{"ts": "2026-04-17T10:00:02Z", "direction": "in", "user_id": "U-OTHER", "text_sha256": "...", "action": "refused-unauthorized-user"}

// proactive notification
{"ts": "2026-04-17T10:01:00Z", "direction": "out", "user_id": "U...", "text_sha256": "...", "action": "notify-review-clean", "pr_num": 501, "repo": "owner/repo"}
```

Never log `text` itself — Slack messages can contain tokens, PR bodies can contain customer data. Hashing preserves "did the same text come in twice?" signal without ever persisting the content.

### Alternatives considered

- **Slack Events API over HTTP (webhooks) instead of socket mode.** Rejected: needs a public HTTPS endpoint with a stable URL. Conflicts with Fly's `auto_stop = suspend` pattern (Machine may be asleep when Slack POSTs and Slack's retry window is short). Socket mode is explicitly designed for this class of bot.
- **Custom slash commands (`/hydra-status`).** Rejected: workspace-level slash commands require Slack workspace admin every time we add or change a command, and they compete with other workspace-level commands. DM-text parsing is self-contained to this one bot / one user.
- **Block Kit rich UI (buttons, modals).** Rejected for Phase 1: more Slack surface area to get right, more places the bot can break silently. Plain markdown replies are predictable and copy-pasteable.
- **A separate `hydra-slack-notifier` process, split from the handler.** Rejected: two processes = two restart lifecycles, two audit logs, two places the `enabled` flag has to be checked. One process with a handler coroutine + a notification coroutine is simpler and still testable.
- **Register the bot as an MCP client eagerly (before #72 lands).** Rejected: bot would have to mock or stub the MCP transport, and then rewrite it once the server lands. Phase-gate with `PHASE1-FALLBACK` is cleaner.

## Test plan

- **Self-test: `slack-config-example-valid` (kind: script).** jq-validates `state/connectors/slack.json.example` exposes the required fields: `enabled`, `bot_token_env`, `app_token_env`, `allowed_user_id`, `default_channel`, `notification_poll_interval_sec`, `repos`. No live Slack API test — that'd need a real workspace. The self-test lands in `self-test/golden-cases.example.json` alongside the other script-kind cases, per the runner's existing dispatch.
- **Manual smoke test (documented in runbook).** Create a dev Slack app, install to a test workspace, set `enabled: true` + real tokens in env, run `./bin/hydra-slack-bot` locally, DM it `status`, verify response. Walk the full runbook once end-to-end before merging.
- **Negative-path manual tests.** Same local setup: (a) wrong user ID DMs the bot → bot silently ignores + logs `refused-unauthorized-user`; (b) unset `HYDRA_SLACK_BOT_TOKEN` → bot exits 1 with clear log line; (c) `enabled: false` in config → bot exits 0 with clear log line.
- **Notification loop unit smoke.** Run the polling thread against a fake `gh pr list` output (e.g. mock via `PATH` shim or a fixture) once, verify it produces one DM per unseen PR, verify the seen-set persists across restart.

The self-test is what CI gates on. The manual smoke + negative-path checks are what the merging operator runs before flipping `enabled: true` in production config.

## Risks / rollback

- **Credential exposure in audit logs.** Mitigated by never logging raw text — only SHA-256 prefixes. If this contract is ever relaxed, the `_description` block in `slack.json.example` calls it out.
- **Slack API outage silently disables notifications.** Bot keeps running but stops delivering. Mitigation: every poll-loop iteration emits a heartbeat log line; if the operator sees no heartbeat for N minutes in `fly logs`, they know.
- **Duplicate notifications on bot restart.** Mitigated by `state/slack-notified-prs.json` persisting the seen-set. Bounded at 500 entries (LRU) so the file stays small.
- **Rate limiting from polling too aggressively.** Mitigated: `notification_poll_interval_sec` has a 30s floor enforced in code, and `gh pr list --label commander-review-clean` is a single API call per repo (O(repos), not O(PRs)).
- **Bot becomes the accidental merge path.** Mitigated by deliberate `NotImplemented` stubs for `merge` / `reject` / `retry`. If a future follow-up wires them, it must go through the commander review gate + `state/connectors/mcp.json:authorized_agents[]` scope check, not a shortcut.
- **Rollback.** Flip `enabled: false` in `state/connectors/slack.json` → bot exits cleanly on next restart. No schema migration, no state cleanup. To fully remove: delete the config file and `logs/slack-audit.jsonl`.
